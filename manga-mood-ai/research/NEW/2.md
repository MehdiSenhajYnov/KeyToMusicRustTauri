## The Narrative State Accumulator (NSA)

Your 4-description sliding window treats context as a flat buffer of recent observations. The fundamental problem isn't that you have *too little* context — it's that context in its current form is **unstructured, equally-weighted, and amnesic past 4 pages**. A description of page N-4 gets exactly the same treatment as N-1, there's no encoding of arc trajectory, and everything before N-4 is gone.

The solution is to replace the flat window with a **structured narrative state object** that accumulates across the entire chapter, maintained by a cheap text-only VLM call between your two existing steps.

---

### Architecture

```
For each page N:

  Step 1: describe_page(image_N) → description_N
    [UNCHANGED — VLM sees image, produces factual description]
    [BUT: new prompt targeting ~150-200 tokens, narrative-focused — see below]

  Step 1.5: update_narrative_state(state_{N-1}, description_N) → state_N
    [NEW — text-only VLM call, no image, ~1-2s]
    [Input: previous state JSON + new description]
    [Output: updated state JSON]

  Step 2: classify_with_state(image_N, state_N, description_{N-1}) → mood + intensity
    [MODIFIED — replaces 4 descriptions with state + 1 description]
```

Three inferences per page. Step 1.5 is text-only so it's fast (~1-2s). Total: ~10-13s, within your acceptable window.

### The narrative state schema

```json
{
  "pages_seen": 15,
  "current_arc": {
    "mood": "tension",
    "intensity_trend": "escalating",
    "pages_in_arc": 5,
    "trigger": "protagonist challenged by rival, fight escalating"
  },
  "previous_arc": {
    "mood": "comedy",
    "length": 4,
    "ended_because": "sudden interruption by antagonist"
  },
  "active_narrative_threads": [
    "protagonist doubts own ability",
    "teammate injured and watching from sidelines",
    "rival revealed hidden technique"
  ],
  "emotional_valence": "negative-escalating"
}
```

This is ~100-150 tokens. Compare to your current 4 descriptions at ~2500-3000 tokens.

### Why this is NOT the same as V10/V11

V10 and V11 failed because they **compressed descriptions** — taking rich text and squeezing it into less rich text. Lossy compression. The signal-to-noise ratio stayed the same or got worse.

The NSA doesn't compress descriptions. It **extracts a different kind of information entirely**: arc structure, trajectory, narrative threads. A description says "character is crying, dark shading, rain, close-up on tears." The state says "we're 5 pages into an escalating tension arc triggered by a confrontation." These are orthogonal signals. The classifier gets the description of the *immediately preceding page* for local texture, and the state for global narrative positioning.

The V11 summary tried to answer "what happened recently?" The state answers "where are we in the story's emotional structure?" — which is what the classifier actually needs to resolve ambiguous pages.

### Why this addresses each documented failure mode

**Early chapter (pages 7-8, insufficient context):** The flat window has only 1-2 descriptions here. The state has seen all 7-8 pages and has already built up an arc object. Even "pages_in_arc: 3" with "intensity_trend: building" gives the classifier more to work with than 2 raw descriptions.

**Arc transition lag (page 23):** This is where the state wins hardest. Your 4-description window at page 23 contains pages 19-22, which are all tension. The classifier sees tension everywhere and misses that page 23 is the sadness break. But the state update at page 23 would detect the mood shift in the description ("character collapses, crying, defeat") against the current arc ("tension, escalating"). The state update prompt explicitly handles this: if the new description's emotional content diverges from `current_arc.mood`, record a transition. So `state_23` would contain:

```json
{
  "current_arc": {
    "mood": "sadness",
    "intensity_trend": "sudden onset",
    "pages_in_arc": 1,
    "trigger": "protagonist defeated, emotional breakdown"
  },
  "previous_arc": {
    "mood": "tension",
    "length": 5,
    "ended_because": "protagonist lost the confrontation"
  }
}
```

Now the classifier sees: image (crying character) + state (brand new sadness arc, just broke from tension) + previous description (the tension climax). This is *exactly* the information a human reader uses to classify this as sadness, not tension.

**Context contamination (page 34, isolated peaceful):** Your current 4-description window drowns the peaceful page in sadness descriptions. The NSA feeds only *one* previous description (N-1) instead of four. The image shows peaceful content. The state says "sadness arc" but the classifier has much more room to weigh the image because it's not fighting against 4 descriptions worth of sadness evidence. One state field saying "current_arc: sadness" is weaker than 4 paragraphs of sadness imagery descriptions — the visual evidence can actually override it.

**Stubborn misclassification (page 31):** If the model is wrong regardless of approach, the state won't fix it. But reducing context noise might help in borderline cases.

### Prompt designs

**Step 1 (describe — revised for narrative focus):**

```
Describe this manga page in 2-3 sentences. Focus on:
- What characters are doing and feeling (expressions, body language, actions)
- What dramatic event or moment is depicted
- The visual atmosphere (lighting, intensity, tone)

Do NOT describe panel layout, text bubble positions, or drawing technique.
Be specific about emotions and actions, not generic.
```

Target: ~150-200 tokens instead of ~500-800. This lets you keep 1 full description in the classify step without context pressure, and it focuses descriptions on the signal the classifier actually uses. You haven't tested description prompt optimization — this is a free variable in your design space.

**Step 1.5 (update narrative state):**

```
You are a narrative tracker for a manga chapter. Given the current tracking state and a new page description, update the state.

CURRENT STATE:
{state_json}

NEW PAGE DESCRIPTION (page {N}):
{description_N}

UPDATE RULES:
- If the page's emotional content matches current_arc.mood, increment pages_in_arc and update intensity_trend
- If it clearly differs, move current_arc to previous_arc and start a new current_arc
- Keep active_narrative_threads to max 3 items, replacing oldest when adding new
- Set emotional_valence to one of: positive-stable, positive-escalating, negative-stable, negative-escalating, neutral, transitioning

Output ONLY the updated JSON, no explanation.
```

This is a structured fill-in-the-blanks task. Small models handle constrained JSON updates well — it's not open-ended generation or summarization, which is why it should outperform V11's free-form summary approach.

**Step 2 (classify — revised):**

```
You are classifying the mood of this manga page for a soundtrack system.

NARRATIVE CONTEXT:
{state_json}

PREVIOUS PAGE:
{description_{N-1}}

AVAILABLE MOODS: {mood_list}
Each mood has intensity 1 (subtle), 2 (moderate), 3 (strong).

<think>
Look at the image carefully. Consider:
1. What do you see? (characters, actions, expressions, atmosphere)
2. How does this connect to the narrative context above?
3. Is this a continuation of the current arc, or a transition to something new?
4. Which mood best fits both what you see AND the story context?
</think>

Answer with exactly: MOOD INTENSITY
Example: sadness 3
```

Note the explicit `<think>` block. **Are you using thinking mode on Qwen3.5-VL 4B?** Your document shows the 2B thinking model at 100% on isolated images, then you switched to 3.5-VL 4B for sequential work — but doesn't mention whether thinking was enabled. Qwen3.x models support thinking via `<think>` tags in the prompt. If you haven't enabled it on the 4B, this alone could be a significant accuracy bump before any architectural changes.

### Expected impact per error type

| Error type | Current cause | NSA fix mechanism | Confidence |
|---|---|---|---|
| Early chapter | Only 1-2 descriptions | State accumulates from page 1 | High |
| Arc transition | 4-description lag | State detects transitions explicitly | High |
| Context contamination | 4 descriptions overpower image | Only 1 description + compact state | Medium-high |
| Stubborn | Unknown | Reduced noise, possibly helps | Low |

Conservative estimate: fixing 3-4 of the 9 errors gets you to 25-26/31 = ~81-84% strict. Fixing 5+ gets you into 85%+ territory. The arc transition and early chapter fixes are high-confidence; context contamination reduction is the swing factor.

### Implementation plan

**Day 1:** Implement the new description prompt (Step 1, shorter/narrative-focused). Run benchmark with the existing V6 pipeline but new descriptions. This isolates whether description quality matters. If accuracy drops, the verbose descriptions carry signal the shorter ones don't — revert and keep the originals.

**Day 2:** Implement the narrative state object and the Step 1.5 update call. Start with a hardcoded initial state for page 1. Run the full 3-step pipeline on the benchmark. Compare against V6.

**Day 3:** If thinking mode isn't already enabled on the 4B, enable it (add `<think>` prompting in Step 2). Benchmark again. This might be the highest-ROI single change.

**Day 4:** Tune. The state schema fields, the update rules, the balance between state and description in the classify prompt — these all have room for iteration. But the architecture itself should show whether the approach has legs by Day 2.

### Failure modes of this approach

**State drift:** If the text-only state update makes bad arc decisions, the state accumulates errors over the chapter. Mitigation: the state is relatively simple (one arc object, one previous arc), and the classify step still sees the image + one description, so a noisy state is less damaging than noisy mood labels (your experiment A). Descriptions are factual, so the state update inputs are grounded.

**4B model can't do structured JSON updates reliably:** Possible, but the update is highly constrained. If the model garbles the JSON, you can parse-and-fix with a simple validator that resets to the previous state on parse failure. Degraded gracefully to the equivalent of "no state update this page."

**State doesn't actually carry useful signal:** If the classifier doesn't benefit from knowing "we're 5 pages into a tension arc," the whole approach adds latency for nothing. This is why Day 2 benchmarking is critical — you'll know immediately.
